## Repository Purpose

This is a personal learning repository containing markdown files documenting various technical concepts, tools, theories, and technologies. The repository serves as a knowledge base for studying and referencing different tech topics.

## Repository Structure

The repository is organized as a collection of markdown files. When creating or organizing content:

- Use clear, descriptive filenames that reflect the topic (e.g., `docker-networking.md`, `react-hooks.md`, `database-indexing.md`)
- Consider organizing files into topic-based directories as the collection grows (e.g., `web-development/`, `databases/`, `devops/`, `algorithms/`)
- Use consistent markdown formatting across all files

## Working with Content

When creating or editing markdown files in this repository:

- Use proper markdown hierarchy with clear headings (# for title, ## for main sections, ### for subsections)
- Include practical examples and code snippets where relevant
- Add links to official documentation or authoritative sources when referencing specific technologies
- Keep content concise but comprehensive enough to serve as a reference

## Writing Style & Tone

All content should be written as if it's part of a well-crafted technical book. Follow these guidelines:

- **Conversational and engaging** — Write like you're explaining to a curious friend, not writing a textbook. Use "you" and "we" freely.
- **Hook the reader** — Every chapter/file should open with a compelling question, a surprising fact, or a relatable scenario that pulls the reader in.
- **Bridge between chapters** — End each file with a teaser or transition that makes the reader want to continue to the next topic.
- **Explain jargon immediately** — Never use a technical term without explaining it the first time. Assume the reader is smart but encountering the concept for the first time.
- **Use analogies** — Compare abstract concepts to everyday things (e.g., attention as "a search engine", residual connections as "a highway").
- **ASCII diagrams liberally** — Use text-based diagrams, tables, and visual layouts to illustrate architecture, data flow, comparisons, and processes. Visuals break up walls of text and aid understanding.
- **Show, don't just tell** — Use concrete step-by-step examples with actual numbers to walk through how an algorithm or mechanism works, not just describe it abstractly.
- **Progressive complexity** — Start simple, layer on complexity. Let the reader build intuition before introducing the formal version.
- **Keep paragraphs short** — 2-4 sentences max. Dense walls of text are hard to read.
- **Highlight key takeaways** — End each chapter with a summary table or bullet list of the most important points.
- **No unnecessary formality** — Avoid phrases like "it should be noted that" or "one might consider". Just say it directly.

## File Naming Conventions

- Use lowercase with hyphens for file names (kebab-case)
- Be descriptive: `kubernetes-deployment-strategies.md` not `k8s.md`
- Group related topics with common prefixes when appropriate
